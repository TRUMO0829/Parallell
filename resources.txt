/**
 * Sequential LSD Radix Sort — Base 256
 * Based on CLRS "Introduction to Algorithms"
 *
 * Key design decisions:
 *   - LSD (Least Significant Digit first) ensures stability across passes
 *   - Base 256: process 8 bits per pass → 4 passes total for uint32_t
 *   - Counting sort used as the stable subroutine each pass
 *   - Two alternating buffers to avoid repeated allocation
 *
 * Complexity: O(d * (n + b))  where d=4 passes, b=256 buckets
 * Space:       O(n + b)


/**
 * Parallel LSD Radix Sort — Base 256, POSIX Threads
 * Inspired by: Obeya et al., "Theoretically-Efficient and Practical
 *              Parallel In-Place Radix Sorting" (SPAA '19, MIT CSAIL)
 *
 * ┌─ Algorithm (one pass over byte `shift` of each uint32_t) ─────────────┐
 * │ Phase 1 (parallel) : each thread counts a local histogram[256]        │
 * │                      over its contiguous chunk of the array.           │
 * │ Barrier                                                                │
 * │ Phase 2 (thread 0) : global prefix-sum over all local histograms      │
 * │                      produces a per-(thread,bucket) scatter offset.   │
 * │ Barrier                                                                │
 * │ Phase 3 (parallel) : each thread scatters its chunk to the output     │
 * │                      buffer using its pre-computed offsets.           │
 * │ Barrier → swap src/dst for next pass                                  │
 * └───────────────────────────────────────────────────────────────────────┘
 *
 * After 4 passes (bytes 0→3, LSB→MSB) the sorted result is back in the
 * original vector (4 buffer-swaps → even, so src pointer is restored).
 *
 * Complexity : O(d·(n + B)) work,  O(d·(n/T + B)) span
 * Space      : O(n + T·B)   (one output buffer + local histograms)
 * where d=4 passes, B=256 buckets, T=thread count.
